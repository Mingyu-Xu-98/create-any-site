# 跨职能团队的共享愿景故事构建

## 何时使用
- 团队成员对产品描述存在明显分歧（如业务期望AI推荐 vs 开发设想VB界面）
- 需在战略领导者、管理者、技术专家等不同听众间建立共同理解
- 需解释颠覆性、违反直觉或缺乏量化数据支持的创意
- 需避免政治化争论，聚焦用户价值与可行性

## 如何执行
1. **引导各方讲述初始愿景故事**：
   - 邀请各团队以叙事形式（含人物、情境、行为、结果）描述理想产品体验
   - 避免仅列出功能或技术要求
2. **对比故事并定位冲突点**：
   - 并列展示各故事，明确分歧所在
   - 识别核心诉求（如“减少操作”、“多媒体呈现”、“技术可行”）
3. **编织统一共享故事**：
   - 整合各方有效元素
   - 包含清晰用户视角和行为路径
   - 确保情节可被全体理解和认同
4. **动态调整信息密度**：
   - **过多细节**：听众无事可做，感到无聊
   - **过少信息**：听众构建出与你意图不同的故事
   - 持续观察肢体语言、表情、评论，判断故事一致性
5. **针对不同听众类型调整策略**：
   - **战略领导者**：连接愿景，展示新趋势
   - **管理者**：聚焦决策支持，推动目标达成
   - **技术专家**：提供可信细节，关联可解决的问题
6. **验证共享故事有效性**：
   - 有明确要点
   - 能开启新思路
   - 展示具体用户行为而非仅依赖数据
   - 为研究数据赋予人性化面孔
