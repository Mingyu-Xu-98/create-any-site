# 用户体验叙事设计框架

## 何时使用
- 正在进行用户体验设计活动（输入收集、创意激发、设计测试、想法分享）
- 需向利益相关者传达用户情境、产品价值或设计决策
- 报告目标是将设计与用户行为/需求关联，并提升受众理解与共鸣
- 不适用于纯数据报告、无用户参与的后台系统开发或非叙事性技术文档

## 核心原则
1. **故事三角关系**：始终明确讲述者、故事本身与听众三者的动态互动。先确定“为什么讲”和“讲给谁”，再选择细节、语言和情感基调。
2. **贯穿设计流程**：将用户故事作为连接用户研究与设计输出的桥梁，用于暴露认知缺口、激发共情并验证方案。
3. **留白与想象**：故事应激活听众想象力，促使其在脑海中填补细节，而非单向灌输信息。

## 推荐叙事结构
根据目标选择以下结构之一：
- **Now-Then-Now**：展示角色与世界关系的延续性或变化性（例：新技术如何以新方式满足旧有基本需求）
- **处方式（Prescriptive）**：采用“Given X, When Y, Then Z”模板，确保因果链合理
- **英雄式（Hero）**：将产品/团队隐喻为拯救用户的英雄
- **由熟至生（Familiar to foreign）**：从熟悉起点逐步引入复杂情境

## 评估标准
使用五维度判断故事结构是否合适：
- **覆盖性**：是否涵盖所有事实，包括不便之处？
- **连贯性**：逻辑是否合理可信？
- **契合度**：是否自然贴合事实？
- **独特性**：解释是否具有说服力？
- **想象空间**：是否为受众留出填充细节的空间？

## 执行要点
- 故事必须直接支撑报告的核心论点
- 优先选择易于复述的结构
- 在书面故事中，明确内嵌结论或通过框架点出要点
- 进行外部审阅，避免作者脑补导致读者误解
